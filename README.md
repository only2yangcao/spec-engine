# Spec Engine - AI 工程自动交付框架

Spec Engine 是一个 AI 驱动的软件工程自动交付框架，通过多角色协作实现从需求规格到代码交付的全流程自动化。

## 核心概念

框架通过定义专业化的 AI 角色，模拟真实的软件开发团队协作流程：

| 角色 | 职责 | 文件 |
|------|------|------|
| 项目经理 | 流程管理、任务分配、进度把控 | `prompts/AGENTS.md` |
| 系统架构师 | 需求理解、工程化任务拆解 | `agents/system-architect.md` |
| 测试工程师 | 测试用例设计、测试验证 | `agents/qa.md` |
| 研发工程师 | 编码实现、单元测试 | `agents/code_developer.md` |
| 代码审核员 | 代码质量审查、测试执行 | `agents/code_reviewer.md` |

---

## 特性汇总

### 一、项目管理视角

| 特性 | 说明 | 实现方式 |
|------|------|---------|
| **多版本管理** | 支持并行多个需求版本的研发，每个版本独立隔离 | `.spec-versions/v{NNN}-{name}/` 目录结构 |
| **断点恢复** | 流程中断后可从断点继续，无需从头开始 | Step0 检测 state.json，精确定位 currentStep/currentTask |
| **状态追踪** | 实时记录流程进展和各任务状态 | `state.json` 统一管理，仅项目经理可修改 |
| **审查记录** | 完整保留所有人工审查的决策历史和反馈意见 | `report.json` 记录每次审查的时间、决策、反馈 |
| **元数据管理** | 记录创建时间、更新时间、中断原因等 | `state.json.metadata` 字段 |
| **强制重启** | 支持灵活重置某个版本的流程 | 删除版本目录或重置 state.json |

---

### 二、流程管理视角

| 特性 | 说明 | 实现方式 |
|------|------|---------|
| **8 步标准化流程** | 从需求初始化到项目汇报的完整标准化流程 | Step0-Step8，线性推进 + 循环迭代 |
| **文件驱动协作** | 所有角色通过读写约定文件协作，无直接交互 | 各环节输入输出文件契约明确 |
| **自反馈循环** | 多层内置自查循环，质量不达标不进入下一环节 | 任务拆解(3轮)、测试用例(指标达标)、编码(双Check)、审查修复(2轮) |
| **3 个人工决策节点** | 关键节点由人类把控，确保方向正确 | 节点1(任务拆解)、节点2(测试用例)、节点3(最终交付) |
| **任务 DAG 管理** | 显式定义任务依赖关系，形成有向无环图 | `task-report.md` 中的依赖关系图 |
| **串行任务执行** | 按依赖顺序逐一完成任务，前一个任务通过后才开始下一个 | Step6 循环执行每个任务 |
| **3 种审查决策** | 每个审查节点可选择通过、不通过或终止 | 通过→推进；不通过→返回修改；终止→回退结束 |

---

### 三、质量管理视角

| 特性 | 说明 | 关键指标/约束 |
|------|------|--------------|
| **任务拆解质量控制** | 深模块原则，控制变更范围 | ≤ 200 行 diff、≤ 5 文件、依赖有效 DAG |
| **双重编码自检** | 函数级 + Commit 级双重检查 | Check A(编译+单元测试)、Check B(静态分析+规范) |
| **修复预算控制** | 限制修复次数和范围，防止无限发散 | 编译无限修复、测试最多3次、Commit修复最多2轮 |
| **不确定性协议** | 分级处理模糊需求，不随意猜测 | 确定→实现、可推断→INFERRED、不确定→SPEC_UNCLEAR、新依赖→NEEDS_HUMAN |
| **8 维度代码审查** | 全方位质量检查 | 正确性、安全性、性能、可读性、架构设计、测试、规范、依赖 |
| **3 级问题分级** | 明确优先级，分级处理 | 🔴 Must Fix(阻塞)、🟡 Should Fix(建议)、🟢 Nice to Have(可选) |
| **审查修复循环** | 持续迭代改进 | 每个任务最多 2 轮修复 |
| **5 维测试覆盖** | 全方位测试场景 | 功能、数据、流程、非功能、集成 |
| **测试硬性指标** | 最低覆盖率要求 | 需求覆盖率 100%、边界值覆盖率 100%、异常场景覆盖率 ≥ 85% |
| **测试用例冻结** | 审查通过后锁定，防止随意变更 | 仅 Step4 可修改，其他阶段需人类同意 |
| **测试强度量化** | 可量化的覆盖质量 | 测试强度 = (实际执行/匹配可执行) × 100% |

## 断点恢复机制

框架支持从任意断点处恢复执行，无需从头开始。

### Step0: 版本检查（断点检测）

每次启动时，项目经理首先执行 Step0 检查：

```
检查 .spec-versions/ 目录
    ↓
存在未完成版本?
    ├─ 是 → 读取 state.json，判断当前进度
    │         ↓
    │      定位到中断时的 Step/Task
    │         ↓
    │      跳过已完成步骤，从断点恢复
    │
    └─ 否 → 要求用户提供需求规格，开始新版本
```

### 断点恢复场景

#### 场景 1: Step2 任务拆解中断

```
中断位置: Step2 进行中（第 2 轮自查）
    ↓
恢复动作:
  1. 读取 tasks/ 目录下已生成的文件
  2. 检查 task-report.md 是否存在
  3. 询问用户: "发现未完成的任务拆解，是否继续？"
  4. 继续 → 从第 2 轮自查继续
  5. 重新开始 → 清空 tasks/，重新执行 Step2
```

#### 场景 2: Step6 某个任务编码中断

```
中断位置: Step6 Task-003 Phase1 编码中
    ↓
恢复动作:
  1. 读取 state.json，确认 Task-001/002 已完成
  2. 检查 git status / 未提交的代码变更
  3. 检查是否有 BLOCKED-A/BLOCKED-B 标记
  4. 展示当前状态给用户确认
  5. 从 Task-003 的编码阶段继续
```

#### 场景 3: 人工审查后返回修改

```
中断位置: Step3 审查不通过，返回 Step2 修改
    ↓
恢复动作:
  1. 读取 report.json 中的审查反馈
  2. 展示上次审查意见给系统架构师
  3. 根据反馈修改任务拆解
  4. 重新进入 Step3 审查
```

### 断点恢复的文件检查清单

| 步骤 | 检查文件 | 恢复判断 |
|------|----------|----------|
| Step2 | `tasks/task-*.md`<br>`tasks/task-report.md` | task-report.md 存在 → Step2 已完成<br>部分 task-*.md 存在 → 执行中 |
| Step4 | `tests/Test*.java`<br>`tests/test-report.md` | test-report.md 存在 → Step4 已完成<br>部分 Test*.java 存在 → 执行中 |
| Step6 | `reviews/review-task-*.md`<br>`reviews/review-report.md` | review-task-N.md 存在 → Task-N 已审查<br>review-report.md 存在 → Step6 已完成 |
| 通用 | `state.json` | currentStep/currentTask 字段 → 精确断点位置 |

### 状态机定义

```
Step 状态:
  ├─ pending     (待执行)
  ├─ in_progress (执行中)
  ├─ reviewing   (人工审查中)
  └─ completed   (已完成)

Task 状态:
  ├─ pending     (待执行)
  ├─ coding      (编码中)
  ├─ reviewing   (审查中)
  ├─ fixing      (修复中)
  └─ completed   (已完成)

审查决策:
  ├─ approved    (通过)
  ├─ changes_req (需修改)
  └─ terminated  (终止)
```

### 强制重启

如需强制重新开始某个版本：

```bash
# 删除指定版本目录
rm -rf .spec-versions/v001-user-auth/

# 或保留文件，仅重置状态
# 编辑 .spec-versions/v001-user-auth/state.json
# 将 steps 重置为 Step0 之后的状态
```

## 工作流程

```
Step0: 版本检查
   ↓
Step1: 需求初始化
   ↓
Step2: [工程化任务拆解] ← (系统架构师) ──┐
   │                                            │ 自查循环
   └────────────────────────────────────────────┘ (最多3轮)
   ↓
Step3: [人工审查] 节点1
   ↓
Step4: [工程化测试验证] ← (测试工程师) ─────┐
   │                                            │ 自查循环
   └────────────────────────────────────────────┘ (需求/边界/异常覆盖率达标)
   ↓
Step5: [人工审查] 节点2
   ↓
Step6: 编码->代码验收 (循环执行每个任务)
   ├─ Phase1: [工程化编码] ← (研发工程师)
   │   ├─ Check A (函数级): 编码→检查→修复 (无限次编译修复/最多3次测试修复)
   │   └─ Check B (Commit级): 完成→检查→修复 (最多2轮)
   ├─ Phase2: [工程化代码审查] ← (代码审核员)
   └─ Phase3: AI审查
         ├─ 通过 → 下一任务
         └─ 不通过 → 返回 Phase1 修复 (最多2轮)
   ↓
Step7: [人工审查] 节点3
   ↓
Step8: [项目汇报]
```

## 自反馈机制详解

框架内置了多层自反馈循环，确保每个阶段的输出质量达到标准后才进入下一环节：

### 1. 任务拆解自反馈 (Step2)

**角色**：系统架构师  
**循环次数**：最多 3 轮  
**检查项**：
- 需求全覆盖矩阵：所有功能点都有对应任务
- 无遗漏、无冗余
- 任务拆分符合深模块原则
- 每个任务 ≤ 200 行 diff、≤ 5 文件变更
- 依赖关系形成有效 DAG

**输出**：`tasks/task-report.md` 包含任务列表、依赖关系图、覆盖矩阵

---

### 2. 测试用例自反馈 (Step4)

**角色**：测试工程师  
**循环条件**：直到覆盖率达标  
**硬性指标**：
- 需求覆盖率 = 100%
- 边界值覆盖率 = 100%
- 异常场景覆盖率 ≥ 85%

**测试维度**：
- 功能维度：正向/反向/CRUD/默认值
- 数据维度：空值/边界/特殊字符/大数据量/精度
- 流程维度：端到端/中断恢复/回退/顺序依赖
- 非功能维度：性能/安全/兼容性/可用性/可靠性
- 集成维度：接口联调/第三方异常/数据一致性

**输出**：`tests/test-report.md` 包含测试强度、覆盖指标、风险提醒

---

### 3. 编码双重检查 (Step6 Phase1)

**角色**：研发工程师

#### Check A (函数级)
- **触发**：每写完一个函数
- **检查**：编译/类型检查 + 单元测试（正常路径+3边界+异常）
- **修复预算**：
  - 编译错误：无限次修复
  - 函数测试：最多 3 次修复
- **修复半径**：≤ 当前函数内部，不改签名，不碰其他函数
- **超限处理**：标记 `BLOCKED-A`，留 TODO，跳到下一个函数

#### Check B (Commit 级)
- **触发**：一个逻辑单元完成
- **检查**：静态分析/linter + CONVENTIONS 规则 + 接口影响声明
- **修复预算**：最多 2 次
- **修复半径**：≤ 本 commit 涉及的文件
- **超限处理**：标记 `BLOCKED-B`，记录问题，继续后续逻辑单元

#### 不确定性协议
| 情况 | 处理方式 |
|------|---------|
| 确定 | 直接实现 |
| 可推断 | 实现 + 注释 `INFERRED: [依据]` |
| 不确定 | 最简实现 + 注释 `SPEC_UNCLEAR: [方案列表]` |
| 新依赖/接口变更 | 不实现，注释 `NEEDS_HUMAN: [描述]` |

---

### 4. 代码审查修复循环 (Step6 Phase3)

**角色**：项目经理  
**循环次数**：每个任务最多 2 轮修复

**流程**：
```
Phase1 编码 → Phase2 审查 → Phase3 AI审查
                               ↓
                        ┌──────┴──────┐
                        ↓             ↓
                      通过         不通过
                        ↓             ↓
                     下一任务    返回 Phase1 修复
                                    ↓
                              记录到 review-report.md
                                    ↓
                              最多 2 轮循环
```

**审查裁决**：
- **PASS**：全部通过，无 Must Fix
- **PASS_WITH_NOTES**：无 Must Fix，有 Should Fix
- **FAIL**：存在 Must Fix → 必须修复

**问题分级**：
- 🔴 **Must Fix**（必须修复）：Bug、安全漏洞、接口不匹配、语义漂移
- 🟡 **Should Fix**（建议修复）：性能隐患、可维护性差
- 🟢 **Nice to Have**（锦上添花）：命名优化、代码风格

**审查维度**（8 个维度）：
1. 正确性（Correctness）
2. 安全性（Security）
3. 性能（Performance）
4. 可读性与可维护性
5. 架构与设计
6. 测试（Testing）
7. 规范与风格
8. 依赖与配置

将本项目直接安装到目标工程的 `.claude` 目录下（**不要**创建 spec-engine 层）：

```bash
# 在目标工程根目录下
mkdir -p .claude
cd .claude

# 将 spec-engine 的内容复制到 .claude 目录
# agents/、prompts/、skills/ 等直接放在 .claude 下
```

目录结构应该是：
```
your-project/
└── .claude/
    ├── agents/
    ├── prompts/
    ├── skills/
    └── ...
```

## 关键特性

### 1. 任务拆解规则
- **深模块**：每个任务包含一个深模块（接口简单但功能丰富）
- **硬约束**：≤ 200 行 diff、≤ 5 文件变更
- **任务依赖**：标注依赖关系，形成 DAG
- **风险识别**：新依赖/接口变更标记 `NEEDS_HUMAN`

### 2. 编码自律
- **Check A**（函数级）：编译检查 + 单元测试（正常路径+3边界+异常）
- **Check B**（Commit 级）：静态分析 + 规范检查 + 接口影响声明
- **不确定性协议**：确定→实现，可推断→`INFERRED`，不确定→`SPEC_UNCLEAR`

### 3. 代码审查标准
- 🔴 **Must Fix**：阻塞合入的严重问题
- 🟡 **Should Fix**：建议修复的问题
- 🟢 **Nice to Have**：锦上添花的改进

审查维度：正确性、安全性、性能、可读性、架构设计、测试、规范、依赖

### 4. 测试覆盖
- 功能维度、数据维度、流程维度、非功能维度、集成维度
- 需求覆盖率 100%、边界值覆盖率 100%、异常场景覆盖率 ≥ 85%
- 测试用例冻结机制：审查通过后不可随意修改
