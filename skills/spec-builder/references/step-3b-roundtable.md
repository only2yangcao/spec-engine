# Step 3b: SPEC 圆桌审查 → SPEC 终稿 (v1.0)
> ⚠️ 本步骤产出物是「SPEC.md v1.0（终稿）」，不是代码。如果你发现自己正在写实现代码，立即停止并回到本步骤。

**预计耗时**：30分钟 | **人类角色**：🔍 审计者

## 定位：轻量级定向审查

> **默认行为**：使用本文件中定义的轻量级定向审查。不要自动调用 `/ljg-roundtable` skill。
> 仅当用户明确说出以下关键词时才建议切换：「深度圆桌」「深度讨论」「/ljg-roundtable」「拉更多人来聊」。

这是 spec-builder 内置的**轻量级圆桌审查**，聚焦于 SPEC 中的假设和盲区，目标是快速收敛出经过压力测试的 SPEC v1.0。

与 `/ljg-roundtable` Skill 的区别：
- **本步骤** = 目标导向，3-5 位角色，聚焦 SPEC 盲区，产出物是修改建议，追求实操可行性
- **`/ljg-roundtable`** = 开放探索，5-8+ 位角色，追求思想深度和跨领域碰撞，适合哲学级讨论

如果用户在审查中说"我想深入讨论某个议题"或"拉更多人来聊"，引导切换到 `/ljg-roundtable`。

## 为什么要做圆桌审查

Step 3a 的苏格拉底追问是一对一的单线交互，擅长帮你"说清楚你想要什么"。但 SPEC 中最危险的不是"写错了什么"，而是**"漏想了什么"**——隐含假设、未考虑的用户场景、被忽视的技术风险、站不住的商业逻辑。多视角圆桌审查的核心价值就是**暴露盲区**。

## 运作方式

### 1. 选人

根据项目类型，动态选取 3-5 位审查角色。角色从以下池中选取，也可以根据项目特点新增：

| 角色 | 视角 | 典型追问 |
|---|---|---|
| 目标用户代表 | 我作为用户真的需要这个吗？ | "你说的痛点我没感觉到"、"这个流程太复杂了我不会用" |
| 技术魔鬼代言人 | 这在技术上可行/合理吗？ | "你考虑过并发问题吗？"、"这个架构到十万用户会崩" |
| 商业怀疑论者 | 这能赚钱/可持续吗？ | "你的护城河在哪？"、"谁会为此付费？" |
| 竞品产品经理 | 我们已经做了类似的东西 | "你跟我们的差异在哪？真的成立吗？" |
| 意外视角者 | 来自完全不同领域的类比挑战 | "在 [类比领域] 这个问题是这样解决的..." |
| 安全/隐私审查官 | 数据风险和合规问题 | "你存储用户数据吗？GDPR 合规了吗？" |
| 运维/SRE 视角 | 上线后谁来维护？ | "半夜挂了怎么办？监控告警呢？" |

用户也可以**指定添加真实历史/当代人物**作为审查者。此时该人物以其真实思想体系发言。

### 2. 审查流程

a. 主持人宣布审查开始，展示 SPEC v0.1 的核心内容摘要和参会角色列表

b. **逐维度审查**（不是每人固定说一轮——根据讨论动态决定谁该发言）：
   - 用户画像是否足够具体？是否遗漏了关键用户群？
   - Use Case 是否真实？用户真的会这样用吗？
   - MVP 功能集是否真的"最小"？能否继续砍？
   - Non-Goals 是否遗漏了应该明确排除的东西？
   - 技术约束是否合理？有没有更好的方案？
   - 成功标准是否可度量？是否在自欺欺人？

c. 每位审查者发言格式：
   ```
   【角色名】【行动标签】：发言内容
   **简言之**：一句话总结
   ```
   行动标签：`质疑` / `补充` / `反驳` / `认可` / `建议修改`

d. 每轮审查后，主持人综述：
   - 提炼**被暴露的盲区**
   - 生成 **SPEC 修改建议清单**——每条建议标注来源和优先级
   - 提问用户：「以上建议，你接受哪些？拒绝哪些？理由是什么？」

### 3. 用户决策

用户作为**最终决策者**，逐条接受或否决修改建议。AI 根据用户决策更新 SPEC.md。

### 4. 退出条件

- 用户确认 SPEC 修改完毕 → 产出 SPEC v1.0
- 如果审查暴露了根本性问题（比如"这个产品根本没有市场"），主持人建议回溯到 Step 0 或 Step 1

## 产出 Artifact

- **SPEC.md v1.0**（终稿）— 经过多视角压力测试的产品规约
- **圆桌审查纪要** — 被暴露的盲区、修改建议及用户决策记录

## 多模块模式补充

> 以下仅在多模块模式下适用。

在多模块模式的模块级圆桌审查中，审查者需额外关注：
- 本模块与其他模块的**接口边界**是否清晰
- 是否有功能/数据被错误地放在了这个模块（应该在别的模块）
- **本模块引用的共享实体是否与 SHARED_SCHEMA 一致**——如果 SPEC 中提到了跨模块数据需求，需确认已在 SHARED_SCHEMA 中注册
- 跨模块的数据一致性假设是否成立
