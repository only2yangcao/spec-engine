# Step 5: Handoff — 交付与执行委托
> ⚠️ 本步骤产出物是「交付摘要」，不是代码。如果你发现自己正在写实现代码，立即停止并回到本步骤。

**预计耗时**：5分钟 | **人类角色**：🎯 指挥官

## 这一步做什么

将规格包正式交付给下游执行者。Playbook 的认知构建工作到此完成。

## AI 做什么

1. 确认规格包完整性——六项交付物全部就绪
2. 展示交付摘要：
   ```
   ══════════════════════════════════════
   📦 规格包交付摘要
   ══════════════════════════════════════
   项目: [项目名]
   SPEC:         ✅ [N 条 User Story]
   DESIGN:       ✅ [N 个实体 / N 个 API / N 条业务规则]
   ARCHITECTURE: ✅ [技术栈摘要]
   CONVENTIONS:  ✅ [主要约定摘要]
   CONTRACT:     ✅ [验收维度 + 执行约束]
   测试场景:      ✅ [N 个场景]
   ══════════════════════════════════════
   ```

   **多模块模式**额外展示：
   ```
   SHARED_SCHEMA:  ✅ [N 个共享实体]
   接口注册表:      ✅ [M 个接口，全部已校验]
   模块:
     模块 A:       ✅ [SPEC + DESIGN + 测试 + 依赖校验通过]
     模块 B:       ✅ [SPEC + DESIGN + 测试 + 依赖校验通过]
   集成测试:        ✅ [K 个跨模块场景]
   ```
3. 提示用户选择下游执行路径（仅做提示，不做执行）：
   - **路径 A：AI Coding** — 将四文件交给 AI Coding 工具（OpenSpec、Cursor、Claude Code 等）
   - **路径 B：Skill 生成** — 将 SPEC 作为功能定义，DESIGN 作为实现规范
   - **路径 C：专业文档生成** — 将 SPEC 作为内容大纲
   - **路径 D：其他** — 根据下游系统的输入规范适配

## 人类做什么

1. 确认规格包完整
2. 选择下游执行路径
3. 将规格包交给下游执行者

## 产出 Artifact

- 完整的规格包（已在 Step 4b 产出，此处仅做确认和交付）
