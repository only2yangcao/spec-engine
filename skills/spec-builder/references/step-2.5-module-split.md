# Step 2.5: 模块拆分 — 大型项目的认知分治
> ⚠️ 本步骤产出物是「全局 SPEC（骨架版）+ 全局 ARCHITECTURE + 共享数据模型 + 模块执行计划」，不是代码。如果你发现自己正在写实现代码，立即停止并回到本步骤。

**预计耗时**：30分钟-1小时 | **人类角色**：🎯 指挥官

## 触发条件

Step 2（竞品分析）完成后，AI 根据意图清单的复杂度判断是否需要模块拆分。检测以下信号：

| 信号 | 权重 |
|---|---|
| 意图清单中 Use Case ≥ 5 个 | +2 |
| 涉及 2 个以上独立数据域（如：用户系统 + 内容系统 + 支付系统） | +2 |
| 涉及外部系统集成 ≥ 2 个 | +1 |
| 预估 MVP 功能点 ≥ 10 个 | +1 |
| 用户主动提到"这个项目比较大"或"需要分模块" | +3（直接触发） |

**判断规则**：
- **总分 ≥ 3** → 建议进入 Step 2.5 模块拆分
- **总分 < 3** → 跳过，直接进入 Step 3a（单模块流程）

AI 展示判断依据并建议，用户做最终决策。

## 模块拆分流程

### 第一步：识别模块候选

AI 基于意图清单，提出模块拆分方案。拆分的原则：

1. **按数据域拆**（首选）：每个模块拥有自己的核心数据实体。如果两个功能操作同一组数据，它们属于同一模块。
2. **按用户旅程拆**（辅助）：不同的用户旅程可能构成不同模块。
3. **模块间只通过接口通信**：如果两个候选模块之间有大量双向依赖，考虑合并。

拆分方案的展示格式：

```
📦 模块拆分方案
══════════════════════════════════════

模块 A: [模块名]
  核心职责: [一句话]
  关键 Use Case: UC1, UC3
  核心数据实体: [实体列表]
  对外接口: [其他模块需要调用的能力]

模块 B: [模块名]
  核心职责: [一句话]
  关键 Use Case: UC2, UC4
  核心数据实体: [实体列表]
  对外接口: [其他模块需要调用的能力]
  依赖: 模块 A（需要调用 [具体接口]）

模块 C: ...

══════════════════════════════════════
跨模块关注点:
  - [全局认证/权限]
  - [共享数据/事件总线]
  - [统一错误处理]
══════════════════════════════════════
建议执行顺序: A → B → C（A 是基础，B 依赖 A，C 独立可并行）
```

### 第二步：用户决策

用户审核拆分方案：
- 调整模块边界（合并/拆分/重命名）
- 确认执行顺序
- 决定哪些模块在 MVP 中做、哪些推迟

### 第三步：识别共享实体 → 生成 SHARED_SCHEMA.md

> 🔑 **核心原则**：跨模块共享的数据实体必须在模块循环之前统一定义，避免各模块各自发明导致数据协议不一致。

模块拆分确认后，AI 分析模块间的数据依赖，识别被 2 个以上模块引用的数据实体，生成 **SHARED_SCHEMA.md**。

**识别规则**：
1. 扫描每个模块的「核心数据实体」和「依赖」字段
2. 被 ≥ 2 个模块引用的实体 → 纳入共享实体
3. 每个共享实体指定唯一的**权威模块**（该实体的"真相来源"，即 CRUD 主控方）
4. 其他模块为**消费模块**，只声明使用哪些字段

**SHARED_SCHEMA.md 格式**：

```markdown
# SHARED_SCHEMA.md — 共享数据模型

> 本文件定义跨模块共享的数据实体。每个实体有且仅有一个权威模块。
> 模块 DESIGN 中引用共享实体时，必须与本文件保持一致。

## 共享实体

### [实体名]（权威模块：[模块名]）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | UUID | ✅ | 全局唯一标识 |
| ... | ... | ... | ... |

**消费模块**：
- [模块 B]（读取字段：id, name, role）
- [模块 C]（读取字段：id, email）

---

### [实体名 2]（权威模块：[模块名]）
...

## 变更规则
- 权威模块修改共享实体字段时，必须同步更新本文件
- 消费模块不得擅自扩展共享实体的字段定义
- 新增共享实体需回到本文件注册，并指定权威模块
```

**展示给用户确认**：

```
📋 共享数据模型（SHARED_SCHEMA.md）
══════════════════════════════════════
共享实体 1: User（权威模块：用户模块）
  字段: id(UUID), email(string), name(string), role(enum), ...
  消费模块: 登录模块(id,email,role), 订单模块(id,name)

共享实体 2: ...
══════════════════════════════════════
```

### 第四步：生成全局 SPEC（骨架版）

模块拆分确认后，生成一份**全局 SPEC.md**，只包含全局信息，不包含模块细节：

```markdown
# [项目名] — 全局 SPEC

## 用户画像
[全局用户画像，所有模块共享]

## 项目愿景
[一句话描述整个项目要做什么]

## 模块架构

| 模块 | 职责 | MVP | 执行顺序 |
|---|---|---|---|
| 模块 A | [一句话] | ✅ | 1 |
| 模块 B | [一句话] | ✅ | 2 |
| 模块 C | [一句话] | ❌ P1 | — |

## 模块依赖关系
[依赖图：谁依赖谁，接口定义在哪个模块的 DESIGN 中]

## 共享数据模型
→ 详见 SHARED_SCHEMA.md

## 跨模块约束
- 全局技术栈: [语言/框架/数据库]
- 全局认证方案: [方案]
- 共享数据规范: [规范]
- 全局 Non-Goals: [不做什么]

## 成功标准（全局）
[全局层面的成功标准]
```

### 第五步：生成全局 ARCHITECTURE.md

同步生成全局架构文档，定义模块间的边界和约束。这比单模块流程中的 Step 4a 提前，因为模块化项目的架构决策必须先于模块细节。

全局 ARCHITECTURE.md 包含：
- 技术栈（全局统一）
- 模块划分与边界
- 模块间通信方式（REST/消息队列/直接调用）
- 部署模式（单体/微服务/模块化单体）
- 共享基础设施（数据库/缓存/消息队列）

## 产出 Artifact

- **全局 SPEC.md（骨架版）** — 只有全局信息 + 模块划分
- **全局 ARCHITECTURE.md** — 模块间架构约束
- **SHARED_SCHEMA.md** — 跨模块共享数据实体定义（权威模块 + 消费模块 + 字段定义）
- **模块执行计划** — 执行顺序 + 每个模块的状态

## 门控检查

```
🚦 门控检查：Step 2.5 → 模块循环
✅ 产出物：全局 SPEC + 全局 ARCHITECTURE + SHARED_SCHEMA + 模块执行计划
✅ 共享实体已识别，每个实体有唯一权威模块
✅ 用户确认：模块拆分方案 + 共享数据模型 + 执行顺序
→ 进入模块循环，从第一个模块的 Step 3a 开始
```

# 模块循环流程

进入模块循环后，按执行顺序逐个处理每个模块。每个模块独立走 Step 3a → 3b → 3c → 4b。

## 模块级进度展示

```
═══════════════════════════════════════
📍 项目进度：[项目名]
═══════════════════════════════════════
✅ 全局 SPEC（骨架）    → 完成
✅ 全局 ARCHITECTURE    → 完成
✅ SHARED_SCHEMA        → 完成（N 个共享实体）

📦 模块 A: [模块名]     → ✅ 完成
   ├─ SPEC.md           → ✅
   ├─ DESIGN.md         → ✅
   ├─ 🔗 依赖校验       → ✅ 无冲突
   └─ 测试场景          → ✅

📦 模块 B: [模块名]     → 🔄 进行中（Step 3c）
   ├─ SPEC.md           → ✅
   ├─ DESIGN.md         → 🔄 进行中
   ├─ 🔗 依赖校验       → ⬜ 待做
   └─ 测试场景          → ⬜ 待做

📦 模块 C: [模块名]     → ⬜ 待做

⬜ 全局 CONVENTIONS     → 待做（合并阶段）
⬜ 全局 CONTRACT        → 待做（合并阶段）
═══════════════════════════════════════
```

## 模块循环中的步骤调整

### 模块 Step 3a（模块 SPEC）

与标准 Step 3a 相同，但有以下调整：
- **输入上下文**：全局 SPEC + 全局 ARCHITECTURE + SHARED_SCHEMA + 前序模块的 DESIGN（用于了解已定义的接口）
- **追问范围**：只追问本模块的 UC 和功能，不重复全局信息
- **产出**：`modules/[模块名]/SPEC.md`

### 模块 Step 3b（模块圆桌审查）

与标准 Step 3b 相同，但审查者需额外关注：
- 本模块与其他模块的**接口边界**是否清晰
- 是否有功能/数据被错误地放在了这个模块（应该在别的模块）
- 跨模块的数据一致性问题
- **本模块引用的共享实体是否与 SHARED_SCHEMA 一致**

### 模块 Step 3c（模块 DESIGN）

与标准 Step 3c 相同，但有以下调整：
- **引用全局 ARCHITECTURE**：技术栈、数据库方案等不在模块 DESIGN 中重复
- **引用 SHARED_SCHEMA**：本模块使用的共享实体必须与 SHARED_SCHEMA.md 的字段定义一致，不得自行修改或扩展
- **定义模块对外接口**：本模块向其他模块暴露的 API/事件/数据结构，写入**接口注册表**
- **声明依赖接口和共享实体**：本模块依赖的其他模块接口和共享实体，也写入接口注册表的依赖段
- **产出**：`modules/[模块名]/DESIGN.md` + 更新接口注册表

### 🔗 依赖校验（每个模块 Step 3c 完成后自动执行）

> **必须执行**：每个模块的 DESIGN 完成后，AI 自动执行依赖校验，不等用户要求。

校验内容：
1. **共享实体一致性**：本模块 DESIGN 中引用的共享实体字段是否与 SHARED_SCHEMA.md 一致
2. **接口参数匹配**：本模块依赖的其他模块接口，入参/出参是否与接口注册表中的定义一致
3. **新增共享需求检测**：本模块是否引入了新的跨模块共享数据，需要追加到 SHARED_SCHEMA

校验结果格式：

```
🔗 依赖校验：模块 B（[模块名]）
═══════════════════════════════════════

📋 共享实体校验：
  ✅ User — 使用字段 (id, email, role) 与 SHARED_SCHEMA 一致
  ✅ Product — 使用字段 (id, name, price) 与 SHARED_SCHEMA 一致

📋 依赖接口校验：
  ✅ 用户模块.获取用户(id:UUID → User) — 入参出参与注册表一致
  ⚠️ 内容模块.获取文章(id:UUID → Post) — 注册表中出参缺少 author_name 字段

📋 新增共享需求：
  🆕 检测到 OrderItem 被模块 B 和模块 C 共同引用 → 建议加入 SHARED_SCHEMA

🔍 冲突汇总：
  ⚠️ 发现 1 处不一致，需解决后继续
  → 建议：在内容模块的接口注册中补充 author_name 字段

═══════════════════════════════════════
```

**冲突处理规则**：
- **✅ 无冲突**：自动继续进入下一步骤（模块 Step 4b 或下一个模块）
- **⚠️ 有不一致**：展示冲突详情，提出修复建议，用户确认后更新相关文件（SHARED_SCHEMA / 接口注册表 / 模块 DESIGN），然后继续
- **🆕 新增共享需求**：提议将新实体加入 SHARED_SCHEMA，用户确认后更新

### 模块 Step 4b（模块测试 + 模块级 CONTRACT）

测试场景仅覆盖本模块内部逻辑 + 模块间接口调用。不生成模块级 CONTRACT——CONTRACT 在合并阶段统一生成。

## 模块间接口注册表（v2 — 带类型契约）

在模块循环过程中，维护一份**接口注册表**（随模块 DESIGN 同步更新）。v2 格式包含完整的类型签名和一致性状态跟踪：

```markdown
# INTERFACE_REGISTRY.md — 模块间接口注册表

> 状态说明：✅ 已校验 | ⚠️ 待校验 | 🔴 冲突

## 模块 A: 用户系统

### 对外接口
| 接口 | 方法 | 入参 | 出参 | 用途 | 消费模块 | 一致性状态 |
|---|---|---|---|---|---|---|
| /api/users/:id | GET | id: UUID | User{id,email,name,role} | 获取用户信息 | 模块B, 模块C | ✅ 已校验 |
| user.created | Event | — | {user_id: UUID, email: string} | 新用户注册事件 | 模块C | ✅ 已校验 |

### 依赖接口
（无外部依赖）

### 依赖共享实体
| 实体 | 权威模块 | 使用字段 | 一致性状态 |
|---|---|---|---|
| User | 自身（权威） | 全部 | ✅ 已校验 |

---

## 模块 B: 登录系统

### 对外接口
| 接口 | 方法 | 入参 | 出参 | 用途 | 消费模块 | 一致性状态 |
|---|---|---|---|---|---|---|
| /api/auth/login | POST | {email,password} | {token, user_id} | 用户登录 | — | ✅ 已校验 |

### 依赖接口
| 来源模块 | 接口 | 入参 | 出参 | 用途 | 一致性状态 |
|---|---|---|---|---|---|
| 用户模块 | /api/users/:id | id: UUID | User{id,email,name,role} | 登录时查用户信息 | ✅ 已校验 |

### 依赖共享实体
| 实体 | 权威模块 | 使用字段 | 一致性状态 |
|---|---|---|---|
| User | 用户模块 | id, email, role | ✅ 已校验 |
```

接口注册表是**活文档**，每完成一个模块的 DESIGN 就更新一次。后续模块在 Step 3c 时读取接口注册表 + SHARED_SCHEMA 作为输入。**依赖校验步骤会更新一致性状态列**。

# 合并阶段

所有模块的循环完成后，进入合并阶段：

## 合并 Step 0：全局一致性审计

> **在生成 CONVENTIONS 和 CONTRACT 之前**，先对所有 artifact 做一次全局交叉校验。

审计内容：
1. **SHARED_SCHEMA 完整性**：所有模块 DESIGN 中引用的跨模块实体是否都已注册到 SHARED_SCHEMA
2. **接口注册表完整性**：所有模块的对外接口和依赖接口是否都已注册，一致性状态是否全部为 ✅
3. **字段漂移检测**：同一共享实体在不同模块 DESIGN 中的字段描述是否一致
4. **孤儿接口检测**：注册表中声明了但未被任何模块消费的接口

审计结果格式：

```
🔍 全局一致性审计
═══════════════════════════════════════
共享实体：N 个，全部已注册 ✅
接口注册表：M 个接口，全部已校验 ✅
字段一致性：无漂移 ✅
孤儿接口：1 个（模块A.user.deleted 未被消费 → ⚠️ 可能是预留接口，标记即可）
═══════════════════════════════════════
审计结论：可以进入合并阶段 / 需要修复 N 处问题
```

## 合并 Step 1：生成全局 CONVENTIONS.md

基于全局 ARCHITECTURE 和所有模块的 DESIGN，统一生成编码规范。

## 合并 Step 2：生成全局 CONTRACT.md

基于全局 SPEC + 全局 ARCHITECTURE + 所有模块的 SPEC/DESIGN/测试，生成统一的交付契约。

CONTRACT 中额外包含：
- **模块间集成验收维度**：除了单模块验收，还要验证模块间接口是否正确集成
- **共享数据一致性验收**：验证各模块对共享实体的使用与 SHARED_SCHEMA 一致
- **交付顺序建议**：建议下游按模块执行顺序逐个实现并集成

## 合并 Step 3：组装最终规格包

```
规格包/
├── SPEC.md              — 全局 SPEC（骨架 + 指向各模块）
├── ARCHITECTURE.md      — 全局架构
├── SHARED_SCHEMA.md     — 跨模块共享数据模型（权威模块 + 字段定义）
├── CONVENTIONS.md       — 全局编码规范
├── CONTRACT.md          — 全局交付契约
├── INTERFACE_REGISTRY.md — 模块间接口注册表（v2 带类型契约）
├── modules/
│   ├── module-a/
│   │   ├── SPEC.md      — 模块 A 规格
│   │   ├── DESIGN.md    — 模块 A 技术设计
│   │   └── tests/       — 模块 A 测试场景
│   ├── module-b/
│   │   ├── SPEC.md
│   │   ├── DESIGN.md
│   │   └── tests/
│   └── ...
└── tests/
    └── integration/     — 跨模块集成测试场景
```

## 门控检查

```

```
