# Step 6: Acceptance — 最终验收
> ⚠️ 本步骤产出物是「验收报告」，不是代码。如果你发现自己正在写实现代码，立即停止并回到本步骤。

**预计耗时**：30分钟-1小时 | **人类角色**：🔍 审计者

## 什么时候进入这步

下游执行者完成实现后，带着产出物回到 playbook 进行验收。

## AI 做什么

按 CONTRACT.md 定义的验收维度，逐层核对下游产出物：

### 第一层：功能完整性（SPEC 层）
- 逐条核对 SPEC.md 中的 User Story
- 每条 User Story 是否有对应的实现？
- 是否有 SPEC 未要求但被实现了的功能（scope creep）？
- 输出：✅ 已实现 / ⚠️ 部分实现 / ❌ 未实现 / 🆕 SPEC 未要求

### 第二层：实现精度（DESIGN 层）
- 逐项比对 DESIGN.md 的定义与实际实现
- 字段名/类型/枚举值是否逐字一致？
- API 端点的请求/响应体是否符合契约？
- 业务规则是否全部且正确实现？
- 输出：✅ 一致 / ❌ 偏离（列出具体偏离点）

### 第三层：风格合规（CONVENTIONS 层）
- 抽检实现是否符合 CONVENTIONS.md
- 命名、目录结构、代码格式是否有系统性偏离？
- 输出：✅ 合规 / ⚠️ 轻微偏离 / ❌ 系统性偏离

### 第四层：测试通过
- 执行 Step 4b 定义的 Given-When-Then 场景
- 输出：✅ 通过 / ❌ 未通过（列出失败场景）

### 第五层：模块间集成验收（多模块模式）

> 仅在多模块模式下执行。

- 逐条校验 INTERFACE_REGISTRY.md 中的接口契约
  - 提供方模块的实际实现是否与注册表中的入参/出参/方法一致？
  - 消费方模块的调用方式是否与注册表匹配？
- 逐条校验 SHARED_SCHEMA.md 中的共享实体
  - 各模块对共享实体的使用字段是否与 SHARED_SCHEMA 定义一致？
  - 是否存在字段漂移（同一实体在不同模块中类型/命名不同）？
- 执行 tests/integration/ 中的跨模块集成测试场景
- 输出：✅ 一致 / ❌ 偏离（列出具体接口/实体偏离点）

### 验收结果处理

```
全部通过 → 验收完成 ✅ → Playbook 流程结束
    │
存在偏离 → 生成偏离报告：
    │     - 哪个维度不通过
    │     - 具体哪一项偏离
    │     - 期望值 vs 实际值
    │     - 建议的修正方向
    │
    └→ 偏离报告交回下游执行者修正
       修正后重新进入 Step 6 验收
```

## 人类做什么

1. 审阅 AI 的验收报告
2. 对偏离项做最终判定：是必须修正，还是可以接受（有些偏离可能是合理的实现优化）
3. 决定验收通过或打回修正

## 产出 Artifact

- **验收报告** — 四层验收结果 + 偏离清单（如有）
- 或：**偏离修正报告** — 交给下游执行者的修正指引

## 偏离报告格式

```
## 偏离报告

### 偏离项 1
- 维度: [功能完整性 / 实现精度 / 风格合规 / 测试通过]
- 具体项: [哪条 User Story / 哪个字段 / 哪条规则 / 哪个测试]
- 期望值: [四文件中的定义]
- 实际值: [实现中的实际情况]
- 严重度: [必须修正 / 建议修正 / 可接受]
- 修正方向: [建议的修正方式]

### 偏离项 2
...
```
