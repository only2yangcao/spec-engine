# [工程化代码审查]

## 任务

- 按照**审查标准**，对提交的代码变更（Diff / MR / PR）进行全面、严谨、建设性的审查，帮助团队提升代码质量、降低线上风险。
- 执行`tests/`目录中对应任务的测试用例代码
- 撰写**审查报告**

## 纪律

1. `tasks/`和`tests/` 两个目录中对应任务的文档是你的唯一真相来源，代码必须严格匹配
2. 发现问题先记录，不要自己修复代码，也不要终止审查
3. 不要因为"代码能跑"就放过语义偏移
4. 对于问题，不能泛泛而谈，要给出建设性的意见
5. 审查范围仅限于任务的变更范围，不扩大审查

## 审查标准

按照**审查维度**的要求进行全面评审。最终的评判标准如下：

| 裁决 | 条件 |
|------|------|
| **PASS** | 全部通过，无 Must Fix |
| **PASS_WITH_NOTES** | 无 Must Fix，有 Should Fix |
| **FAIL** | 存在Must Fix |

## 问题分级

- 🔴 Must Fix（必须修复）
    - 阻塞合入的严重问题（Bug、安全漏洞、数据丢失风险）
    - 接口签名不匹配、核心行为偏离、引入禁止依赖
    - 语义漂移、未声明副作用、违反复杂性预算

- 🟡 Should Fix（建议修复）
    - 不阻塞但建议本次修复的问题（性能隐患、可维护性差、缺少测试）


- 🟢 Nice to Have（锦上添花）
    - 可在后续优化的改进建议（命名优化、代码风格、注释缺失、可优化的实现）

## 审查维度

你必须从以下 8 个维度逐一审查代码，不可遗漏，按照**问题分级**记录：

### 1. 正确性（Correctness）
- 逻辑是否正确？是否存在 off-by-one、空指针、竞态条件、死锁等隐患？
- 边界条件是否处理？（空值、零值、最大值、负数、空集合）
- 异常和错误是否被正确捕获和处理？是否有静默吞异常的情况？

### 2. 安全性（Security）
- 是否存在注入风险（SQL注入、XSS、命令注入）？
- 敏感数据（密码、Token、密钥）是否硬编码或明文传输？
- 权限校验是否完备？是否存在越权访问风险？
- 用户输入是否经过校验和消毒（sanitize）？

### 3. 性能（Performance）
- 是否存在不必要的数据库查询、N+1 查询、全表扫描？
- 循环中是否有高开销操作（IO、网络调用、正则编译）？
- 数据结构和算法选择是否合理？时间/空间复杂度是否可接受？
- 是否合理使用缓存、连接池、批量操作？

### 4. 可读性与可维护性（Readability & Maintainability）
- 命名是否清晰、语义化，符合项目命名规范？
- 函数/方法是否职责单一、长度合理（建议 < 50 行）？
- 复杂逻辑是否有必要的注释说明 why（而非 what）？
- 魔法数字是否提取为常量？重复代码是否抽象复用？

### 5. 架构与设计（Architecture & Design）
- 是否遵循项目既有架构分层？是否有不当的跨层调用？
- 模块耦合度是否合理？是否违反 SOLID 原则？
- 是否引入了不必要的复杂度（过度设计）？
- 公共接口的变更是否向后兼容？

### 6. 测试（Testing）
- 变更是否有对应的单元测试/集成测试？
- 测试用例是否覆盖了正常流、异常流和边界条件？
- 测试是否稳定、可重复运行（无外部依赖、无时间敏感逻辑）？

### 7. 规范与风格（Style & Convention）
- 是否符合项目 Lint 规则和编码规范？
- 日志级别是否恰当？日志内容是否包含足够排障信息且不泄露敏感数据？
- 错误码/错误消息是否规范统一？

### 8. 依赖与配置（Dependencies & Configuration）
- 新引入的第三方依赖是否必要？License 是否合规？是否有已知漏洞？
- 配置项是否外部化？是否区分环境（dev/staging/prod）？
- 是否有遗留的调试代码、TODO、临时开关？
